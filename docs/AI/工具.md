# 工具

| 维度 | IDE类型 | CLI类型 | VSCode扩展类型 |
|------|---------|---------|----------------|
| **启动速度** | 较慢（需启动完整IDE） | 最快（直接终端运行） | 中等（即时加载） |
| **上下文感知** | 最强（完整项目理解） | 中等（依赖传入的文件） | 较强（IDE API+文件） |
| **交互方式** | 面板/对话式 | 纯命令行对话 | 侧边栏/对话框 |
| **跨项目能力** | 一般绑定单个项目 | 强（随时切换项目） | 中等 |
| **资源占用** | 最高 | 最低 | 中等 |

## 使用场景

### CLI命令行

- 快速修复/单文件修改
- 脚本生成/自动化任务
- 跨多个项目快速操作
- 与现有工作流集成（Git Hooks、C Makefile等）
- 服务器上直接操作

#### 优势

轻量、灵活、适合DevOps场景

### VSCode扩展

- 日常编码辅助（智能补全）
- 调试时快速提问
- 与VSCode深度集成（代码跳转、终端等）
- 需要可视化对比/diff的场景
- 不需要离开编辑器的工作

#### 优势

无缝嵌入开发体验，鼠标悬停即可交互

### IDE类型（独立应用）

- 复杂项目重构
- 多文件/多模块系统开发
- 需要深度分析代码库结构
- 项目前期设计/架构讨论
- 长对话上下文理解

#### 优势

完整的索引+上下文+独立的AI交互空间

## 实际选择建议

- 简单任务（单文件、快速修复）→ CLI
- 日常编码（补全、small talk）→ VSCode扩展
- 复杂任务（重构、多文件）→ IDE类型

## 关键差异

- CLI的优势在于**轻量和可脚本化**
- VSCode扩展在于**无感的上下文**
- IDE类型在于**深度理解和完整功能**
